# Beacon v0.1.0 (Alpha)

## 简体中文

Beacon v0.1.0 是项目的首个公开 Alpha 版本。它提供一套本地优先、服务商
无关的协作层，让具备 CLI 操作能力的 AI Agent 能够通过明确的身份、端点、
请求和状态记录在同一项目中协作。

### 主要能力

- 提供本地工作空间、Agent、上下文、对话、请求、投递和状态管理。
- 提供幂等的服务商接入流程，可统一创建或复用 Agent、服务商会话句柄和
  工作空间端点别名。
- 支持 Claude、Codex 和 Hermes 已登记会话的元数据发现与受限激活。
- 默认只发现会话元数据；读取消息片段或更完整历史必须显式启用并受到范围
  限制。
- 提供请求排队、单次投递、守护进程存活检测、租约恢复、运行退避和时间线
  状态查询。
- 提供仅限本机访问的可选 TypeScript Gateway，以及部分 Python CLI bridge
  接口。
- 建立版本一致性、发布卫生、Python wheel 构建和 Windows/Ubuntu、
  Node.js 22/24 持续集成检查。

### 下载与安装

GitHub Release 会提供与 <code>v0.1.0</code> 对应的 ZIP 和 TAR 源码归档。
下载并解压源码后，可在仓库根目录执行：

~~~powershell
py -3.11 -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -e .\python-core
.\.venv\Scripts\beacon.exe --help
~~~

本版本尚不提供 Beacon 桌面应用或 Windows 一键安装程序。CI 中构建的 Python
wheel 只覆盖 Python Core，不能代表包含 Gateway 的完整桌面发行包。

### 安全与隐私

- Beacon 不保存服务商凭据，凭据仅从明确指定的环境变量读取。
- Gateway 拒绝非 loopback 监听地址，不支持 LAN 或公网部署。
- 服务商重连、警告和错误事件不会被误判为 Codex 的最终回复。
- 请通过 [SECURITY.md](../../SECURITY.md) 中的私密渠道报告漏洞，不要在公开
  Issue 中附带凭据、完整对话、会话注册表、数据库或本地日志。

### 已知限制

- 当前版本用于本地实验和受控协作，不是生产级多用户服务。
- Gateway 尚未覆盖完整 Python CLI 能力。
- 共享 JSON 注册表不保证多进程并发写入安全。
- 尚未提供 MCP 接口、桌面 UI、远程 Agent 托管、服务商账号连接器或
  多 Agent 群聊。
- Beacon 不会绕过服务商登录、交互式授权或操作系统沙箱。

### 反馈

Beacon 的早期实现主要由 vibe coding 推动，部分结构或实现可能仍有改进
空间。欢迎提交 Issue、设计建议或 Pull Request；涉及安全问题时请使用
私密报告渠道。

## English

Beacon v0.1.0 is the first public alpha release of a local-first,
provider-agnostic coordination layer for CLI-capable AI agents. It lets
independently running agents collaborate in one project through explicit
identities, endpoints, requests, and observable status.

### Highlights

- Local workspace, agent, context, conversation, request, dispatch, and status
  management.
- Idempotent provider onboarding that creates or reuses agents, provider
  session handles, and workspace endpoint aliases.
- Metadata discovery and bounded activation for registered Claude, Codex, and
  Hermes sessions.
- Metadata-only discovery by default; message snippets and broader history
  require explicit, bounded opt-in.
- Queued and one-shot dispatch, daemon liveness, lease recovery, runtime
  backoff, and timeline status.
- An optional localhost-only TypeScript Gateway with selected Python CLI bridge
  routes.
- Version, release-hygiene, Python wheel, Windows/Ubuntu, and Node.js 22/24 CI
  checks.

### Download And Installation

The GitHub Release provides ZIP and TAR source archives for
<code>v0.1.0</code>. After extracting the repository, run:

~~~powershell
py -3.11 -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -e .\python-core
.\.venv\Scripts\beacon.exe --help
~~~

This release does not include a Beacon desktop application or a one-click
Windows installer. The Python wheel built in CI covers Python Core only and is
not a complete desktop distribution with Gateway.

### Security And Privacy

- Beacon does not persist provider credentials; values are read only from
  explicitly named environment variables.
- Gateway rejects non-loopback bind targets and does not support LAN or public
  deployment.
- Provider reconnect, warning, and error events cannot become final Codex
  responses.
- Report vulnerabilities through the private channel in
  [SECURITY.md](../../SECURITY.md). Do not attach credentials, full
  transcripts, session registries, databases, or local logs to a public issue.

### Known Limitations

- This release is for local experimentation and bounded collaboration, not a
  production multi-user service.
- Gateway does not expose the full Python CLI surface.
- Shared JSON registry updates do not guarantee safe concurrent writes across
  multiple processes.
- MCP integration, a desktop UI, remote agent hosting, provider account
  connectors, and multi-agent rooms are not included.
- Beacon does not bypass provider login, interactive approval, or operating
  system sandbox controls.

### Feedback

Beacon's early implementation was driven largely through vibe coding, so some
structural or implementation decisions may deserve revision. Issues, design
feedback, and pull requests are welcome. Use the private reporting channel for
security-sensitive findings.
